## 继续梳理：把“方法清单”升级为“决策系统”

你这段总结已经覆盖了关键部件。下一步的价值不在于再解释概念，而在于把它们组织成可执行的选择框架：遇到一个真实业务问题时，如何用最小成本选到最稳方案。

---

# 1 核心概念深化：从定义到边界条件

## 1.1 大模型的角色定位

* 大模型擅长：语言理解、生成、归纳、弱规则推理、通用常识。
* 大模型不擅长：可验证事实的实时性、强一致的格式输出、严格受控的业务规则、低延迟高并发的稳定交付。

这决定了后面所有“增强手段”本质都在补足四类缺口：知识缺口、规则缺口、格式缺口、执行缺口。

## 1.2 Prompting：把“需求”变成“约束”

Prompt 的本质不是问问题，而是做三件事：

1. **定义任务边界**：你要它做什么，不做什么。
2. **提供判分标准**：什么叫好答案，什么叫错答案。
3. **规定输出协议**：结构、字段、长度、语气。

当你发现 Prompt 需要不断加长才能稳定时，通常意味着你在用 Prompt 代替系统能力，应该升级到 RAG、微调或 Agent。

## 1.3 RAG：把“事实”外置为可检索资产

RAG解决的不是生成能力，而是“证据供给”。它把“模型脑内记忆”改成“外部证据驱动”，因此：

* 适合：知识不断更新、需要可追溯依据、私域知识问答、减少幻觉。
* 不适合：强格式输出、严格业务流程编排、极低延迟场景。

## 1.4 微调：把“行为模式”固化进模型

微调更像把“示范答案”压进模型参数里，强化特定风格或任务技能，因此：

* 适合：稳定格式化输出、固定风格、意图到指令的映射、低 token 输入。
* 不适合：知识频繁更新、数据难以高质量标注、需要透明引用依据的场景。

---

# 2 RAG vs 微调：把对比表升级为决策树

你已有对比表，建议再加一层“先问三个问题”的判别逻辑：

## 2.1 判别问题 A：知识是否频繁变化

* 频繁变化：优先 RAG
* 相对稳定：微调可行

## 2.2 判别问题 B：输出是否需要严格协议

* 强格式、可直接对接系统：倾向微调
* 自然语言解释为主：倾向 RAG

## 2.3 判别问题 C：你是否需要可审计证据

* 需要给出处、可追溯：倾向 RAG
* 不要求引用，只要结果：微调更合适

你文中一句“效果几乎一样”需要补一层限定：
表面回答相似不等于系统性质相同。RAG强调可更新与可追溯，微调强调低 token 与稳定映射，这两者的工程代价完全不同。

---

# 3 Agent 与多模态：把“能力描述”落到“系统形态”

## 3.1 Agent的本质

Agent不是更聪明的模型，是更完整的软件系统：

* **规划模块**：把目标拆成步骤
* **工具调用**：把语言变成动作
* **记忆系统**：把历史变成可用信息
* **治理机制**：把风险拦在输出前

因此 Agent 适合的不是“回答问题”，而是“完成任务闭环”，例如写周报并发送邮件，拉取数据并生成报表，检索文档并生成合规回复。

## 3.2 多模态的本质

多模态不是花哨功能，而是两类收益：

* 输入侧：降低信息损耗，例如图片、语音直接进入系统
* 输出侧：提高交付效率，例如图表、语音、视频直接输出

把多模态当作“交互与信息采集升级”，不要当作“更强的聊天”。

# 4 第二章的“要耐心、多尝试”等内容：建议做结构升级

这段更像价值宣言。为了让它更像课程结尾的“方法论”，可以改为四条可操作原则：

1. **先小步闭环，再扩能力**
   先让系统在一个场景端到端跑通，再加 RAG、微调、多模态、Agent。

2. **先评测再优化**
   没有指标的优化是自我感动。至少要定义：准确率、引用命中率、格式通过率、延迟、成本。

3. **先治理再上线**
   先做输入审核、输出审核、敏感信息拦截、日志与追溯，再谈规模化接入业务。

4. **先分层再协作**
   产品定义问题，业务定义口径，工程定义系统边界，算法定义模型能力。跨界合作要围绕“统一口径与可验证交付”。

你现在已有：Prompt、RAG、微调、Agent、多模态、多智能体。最缺的一张图是“分层架构”：
